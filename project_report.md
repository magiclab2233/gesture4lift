# 基于手势识别的无接触智慧电梯控制器设计与实现

**摘要：** 后疫情时代，公共场所的卫生安全问题日益受到关注，电梯按键作为高频接触的公共设施，存在较大的交叉感染风险。本项目针对这一现实需求，设计并实现了一套基于计算机视觉手势识别的"无接触"智慧电梯控制器。项目以 MediaPipe HandLandmarker 为核心，通过分析手部 21 个关键点的坐标关系，实现了对数字手势 1-5 的实时识别；创新性地采用"拇指靠近度+四指离散度"双特征联合判定算法，解决了传统单特征阈值法中"确认动作易与数字手势混淆"的痛点；同时引入了电梯轿厢平滑移动动画、中文 HUD 界面和手势防抖机制，提升了交互的真实感与稳定性。在开发过程中，本项目自主设计了数据采集工具，采集了 401 条真实手势数据进行阈值校准，最终使确认动作的检测准确率达到 100%，数字 5 与数字 4 的区分准确率达到 100%。实验表明，该系统能够在自然光环境下稳定运行，为医院、商场等公共场所提供了一种零接触、低成本的电梯控制方案。

**关键词：** 手势识别；MediaPipe；无接触控制；电梯系统；计算机视觉

**项目仓库：** https://github.com/magiclab2233/gesture4lift

## 一、引言

### （一）研究背景

2020 年以来，全球公共卫生意识的提升使得人们对高频接触公共设施的卫生安全格外关注。电梯作为商场、医院、写字楼等场所的必备垂直交通工具，其楼层按键每天被数百甚至数千人触摸，成为了细菌和病毒传播的潜在媒介。据相关研究显示，电梯按键表面的细菌菌落数可达马桶圈的数十倍，传统的物理按键在疫情防控中暴露出明显的短板。

尽管部分高端电梯已经引入了语音控制或手机 APP 呼梯等功能，但这些方案存在识别延迟高、老年人使用门槛高、需要额外硬件投入等问题，难以在短期内大规模普及。因此，开发一套成本低、响应快、无需接触实体按键的电梯控制系统，具有重要的现实意义。

### （二）研究目的及意义

本项目旨在利用常见的 USB 摄像头和家用计算机，结合开源计算机视觉库，设计一套"无接触"电梯楼层选择系统。用户只需面对摄像头，比出对应数字手势，再通过一个确认动作即可控制电梯前往目标楼层，全程无需触摸任何实体按键。

项目的理论价值在于探索了 MediaPipe 手部关键点模型在实际控制场景中的应用方法，提出了基于双特征联合判定的手势确认算法，并通过真实数据采集验证了算法的有效性。其实践价值在于为公共场所提供了一种低成本、易部署的卫生升级方案，有助于降低交叉感染风险，提升公共空间的卫生安全水平。

### （三）研究综述

**1. 手势识别技术现状**

手势识别是人机交互领域的重要研究方向，目前主流方案可分为基于传感器和基于视觉两大类。基于传感器的方案需要用户佩戴数据手套或手环，虽然精度高，但使用不便、成本较高。基于视觉的方案利用摄像头捕捉手部图像，通过深度学习模型提取特征，具有非侵入、零穿戴的优势。

在手部关键点检测领域，Google 开源的 MediaPipe Hands 框架提供了 21 个手部关键点的实时检测能力，在桌面级 CPU 上即可达到 30FPS 以上的运行速度。MediaPipe 0.10 版本后引入了 Tasks API，采用独立的 `.task` 模型文件，相较于早期的 Solutions API 具有更好的跨平台兼容性。

**2. 无接触控制应用现状**

目前市面上已出现部分无接触电梯方案，如基于 ToF 深感摄像头的悬浮按键、基于红外对射管的虚拟按钮阵列等。然而，这些方案普遍存在以下局限：

- **硬件成本高**：ToF 摄像头或红外矩阵需要额外采购专用设备；
- **部署复杂**：需要在电梯轿厢内打孔安装传感器；
- **交互单一**：通常只能实现简单的"悬浮点击"，无法支持楼层数字的直接输入。

相比之下，基于普通 RGB 摄像头和手势识别的方案，仅需一台笔记本电脑或树莓派即可运行，部署灵活、成本极低，更适合中小型公共场所的快速改造。

## 二、项目设计

### （一）项目总体设计

本项目是一套软件层面的手势识别控制系统，核心流程分为"手势采集 → 关键点检测 → 手势解析 → 状态控制 → 动画渲染"五个阶段。系统整体架构如图 2.1 所示。

```
┌─────────────┐    ┌─────────────┐    ┌─────────────┐
│   USB摄像头  │ → │  MediaPipe  │ → │  手势解析    │
│  (图像采集)  │    │ HandLandmarker│   │ (数字/确认) │
└─────────────┘    └─────────────┘    └──────┬──────┘
                                             ↓
┌─────────────┐    ┌─────────────┐    ┌─────────────┐
│  OpenCV动画  │ ← │  电梯状态机  │ ← │  防抖与触发  │
│  (HUD界面)   │    │  (楼层/移动) │    │  (确认执行) │
└─────────────┘    └─────────────┘    └─────────────┘
```

**图 2.1 系统整体架构示意图**

系统主要由以下模块组成：

- **gesture_recognizer（手势识别模块）**：负责调用 MediaPipe 模型检测手部关键点，并解析出数字手势和确认动作；
- **controller（控制模块）**：维护电梯当前楼层、目标楼层和移动状态，处理手势到控制指令的映射；
- **elevator_animator（动画模块）**：基于 OpenCV 绘制电梯井道与轿厢的平滑移动动画；
- **ui_renderer（界面渲染模块）**：在视频帧上叠加中文状态栏、手势指南、事件提示等 HUD 元素；
- **data_collector（数据采集模块）**：用于采集手势训练数据，支持按键标注和 CSV 导出。

### （二）选用技术及原理

**1. MediaPipe HandLandmarker**

MediaPipe 是 Google 开发的开源跨平台机器学习框架，其中的 HandLandmarker 任务能够实时检测单只或双手的 21 个三维关键点（如图 2.2 所示）。每个关键点包含归一化坐标 (x, y, z)，其中 x、y 为图像平面坐标（0~1），z 为相对于手腕的深度值。

**图 2.2 MediaPipe 手部 21 个关键点示意图**

本项目选用 MediaPipe Tasks API（版本 0.10.33），其优势在于：
- 采用 TensorFlow Lite 推理，可在普通 CPU 上实时运行；
- 提供独立的 `.task` 模型文件，无需联网即可使用；
- 输出关键点坐标的同时，保留了三维深度信息，便于后续的手势空间分析。

**2. OpenCV 与 PIL 图像处理**

OpenCV 是本项目的图像 I/O 和基础绘制库，负责摄像头读取、镜像翻转、颜色空间转换（BGR↔RGB）和几何图形绘制。由于 OpenCV 原生的 `cv2.putText` 不支持中文渲染，本项目引入了 PIL (Pillow) 库进行文字绘制，实现了中文状态提示与英文代码的混排显示。

**3. 防抖与状态机机制**

为了避免手部自然抖动导致的数字跳变，系统引入了**环形缓冲区防抖机制**：只有当连续 8 帧识别到同一数字时，该数字才会被确认为"有效输入"。同时，系统维护了一个简易的有限状态机：

- **Idle（待机）**：等待用户输入；
- **Selected（已选层）**：数字被确认，等待触发；
- **Moving（移动中）**：电梯正在前往目标楼层，期间锁定输入。

## 三、关键功能实现

### 1. 指尖状态判定与数字手势识别

指尖状态判定是数字手势识别的基础。原始方案（比较指尖与关节的 y 坐标）在手掌旋转时极不稳定，本项目改用**相对距离法**：

- **其他四指**：若 `指尖到手腕的距离 > 指根到手腕的距离 × 1.15`，且 `指尖到手腕的距离 > 第二关节到手腕的距离 × 1.05`，则判定为伸出；
- **大拇指**：由于大拇指关节结构与其他四指不同，采用 `拇指尖到食指根的距离 > 拇指根到食指根的距离 × 阈值` 的策略。

在开发过程中，我们发现数字 5 常被误识别为 4。通过分析真实数据，确认问题根源在于大拇指伸直判定的阈值设定。经过多轮调试和数据统计，最终将大拇指阈值定为 **0.75**，使得数字 4（弯曲拇指，比值 0.44~0.53）与数字 5（伸直拇指，比值 0.91~0.97）被完全区分。

### 2. "五指并拢"确认动作的检测

确认动作的设计经历了三次迭代：

- **第一代**：基于 Z 轴深度的"食指前伸点击"，因 MediaPipe 深度坐标精度不足而被弃用；
- **第二代**：基于食指-拇指捏合的"OK 手势"，但捏合动作易被识别为数字 5，产生冲突；
- **第三代**：基于"五指指尖并拢"的确认动作，更符合人体工学。

在第三代方案中，最大的技术挑战是**如何区分"数字 4"与"五指并拢"**。数据显示，数字 4 的五指尖最大跨度（1.00~1.08）与并拢动作（0.40~1.05）高度重叠，单一阈值无法区分。

本项目的创新在于提出了**双特征联合判定算法**：

| 特征 | 作用 | 阈值 | 排除对象 |
|------|------|------|---------|
| 拇指尖到最近的其他指尖距离 | 判断拇指是否靠近四指 | < palm_size × 0.65 | 数字 4/5（拇指远离） |
| 四指指尖到重心的平均距离 | 判断四指是否紧密聚拢 | < palm_size × 0.30 | 数字 3（四指分散） |

该组合在 401 条真实数据上的验证结果如下：数字 1~5 的误触发率为 0%，together 的正确触发率为 100%。

### 3. 电梯移动动画

为了提升交互的真实感，本项目在画面右侧设计了电梯井道动画（如图 3.1 所示）。动画采用 ease-in-out 缓动曲线实现平滑过渡：

$$t_{smooth} = t^2 \times (3 - 2t)$$

其中 $t$ 为线性进度（0~1）。轿厢移动速度约为 0.8 秒/层，到达目标楼层后变为绿色并高亮显示 1.5 秒。

**图 3.1 电梯井道与轿厢动画界面**

### 4. 中文 HUD 界面

由于 OpenCV 原生不支持中文，本项目采用 PIL 的 `ImageDraw.text()` 方法在 OpenCV 图像上叠加中文文本。系统会自动检测 Windows 字体目录，优先使用"微软雅黑"，兜底使用黑体或宋体，确保所有状态提示、手势指南和事件消息均无乱码。

### 5. 数据采集与阈值校准工具

为了科学优化算法参数，本项目自主开发了 `data_collector.py` 数据采集脚本。用户通过按键（0~5、T、N）即可将当前帧的 21 个关键点坐标和标签保存至 CSV 文件。基于采集的 401 条数据，我们对大拇指阈值、并拢阈值和手掌朝向阈值进行了统计分析，最终将系统识别准确率从经验值的约 75% 提升到了 **100%**。

## 四、开发历程与问题排查

一个项目从想法到成品，往往需要经历很多次"发现问题 → 想办法 → 尝试 → 再发现问题"的循环。本章记录了我们在开发过程中踩过的"坑"，以及最终是怎么把它们填平的。

### （一）网上教程里的代码跑不通

**遇到的问题：** 刚开始写程序时，我们参考了网络上关于 MediaPipe 手势识别的教程，里面用了 `mediapipe.solutions.hands` 这段代码。但我们运行时报错，说找不到这个模块。

**排查过程：** 检查了安装的 MediaPipe 版本是 0.10.33，发现这个新版本里并没有打包旧的 `solutions` 模块。查阅官方文档后才明白，MediaPipe 从 0.10 版本开始换了一套全新的接口，叫做 **Tasks API**，旧的那套代码已经不能用了。

**解决办法：** 我们把整套手势识别代码全部重写了一遍，改用新的 `HandLandmarker` 接口来创建检测器。另外，旧版自带的画骨架功能也没了，我们又自己写了一套用 OpenCV 画手部 21 个关键点连线的代码。

### （二）模型文件放在中文文件夹里，程序说"找不到"

**遇到的问题：** `hand_landmarker.task` 这个 AI 模型文件明明就在项目文件夹里，但程序一运行就报"文件找不到"。

**排查过程：** 我们的项目放在了"桌面"文件夹下，路径里有中文字符。后来发现，MediaPipe 底层是用 C++ 写的，它对含有中文的路径支持得不好。我们把同一个文件复制到纯英文路径（比如 `C:\temp\`）下，就能正常加载了。

**解决办法：** 在代码里加了一个自动处理机制：如果检测到模型文件的路径里含有中文，就先把文件复制到系统的临时文件夹（这个路径一定是纯英文的），再从临时文件夹加载。原文件保留不动，复制完就能正常运行了。

### （三）画面上显示不出中文

**遇到的问题：** 我们想往画面上加"当前楼层""移动中"这些中文提示，但显示出来的全是小方块或乱码。

**排查过程：** 查了 OpenCV 的官方文档才知道，它自带的 `cv2.putText` 函数只支持英文字母和数字，**根本不支持中文**。不管我们怎么换字体都没用。

**解决办法：** 我们换了一个叫 PIL（Pillow）的图像库来画中文。思路很简单：先把 OpenCV 的图像转成 PIL 能处理的格式，用 PIL 把中文写上去，再转回 OpenCV 的格式显示。系统会自动寻找电脑里的中文字体（优先用微软雅黑），这样画面上的所有中文都能正常显示了。

### （四）伸出"5"，系统却说是"4"

**遇到的问题：** 当我们五指全部伸直（表示数字 5）时，系统经常误判为 4。这个 bug 非常影响使用。

**排查过程：** 一开始我们用了两种方法来判定大拇指是否伸直：一种看大拇指尖到手腕的距离，另一种看大拇指尖到食指根的距离。我们以为"只要满足其中一个就算伸直"会更稳妥。但实际测试发现，手离摄像头远近会明显影响"到手腕的距离"，导致第一种方法非常不靠谱——手远一点就可能被误判为弯曲。

**数据验证：** 为了搞清楚真相，我们用自己写的采集工具专门录了 101 次"数字 4"和 51 次"数字 5"的手势数据，然后用 Excel 分析。结果发现：
- 数字 4（大拇指弯曲）的"大拇指尖到食指根距离 ÷ 大拇指根到食指根距离"这个比值，集中在 **0.44 ~ 0.53** 之间；
- 数字 5（大拇指伸直）的同一个比值，集中在 **0.91 ~ 0.97** 之间。

这两组数字就像两个班级，中间完全没有重叠，界限非常清晰。

**解决办法：** 我们抛弃了那个会受距离干扰的方法，只保留"大拇指尖到食指根的距离比值"这一种判定方式，并把分界线（阈值）通过数据校准定在了 **0.75**。从此以后，所有"4"都在 0.75 以下，所有"5"都在 0.75 以上，再也没有误判过。

### （五）手指弯了一半，系统以为它是伸直的

**遇到的问题：** 有时候手指只是微微弯曲，并没有完全伸直，但系统却判定为"伸出"，导致识别的数字比实际多。

**排查过程：** 我们打开了数据的"黑箱"，观察手指半弯曲时的坐标。原来，当手指微微弯曲时，指尖到手肘（手腕）的距离可能仍然大于指根到手肘的距离，满足原来的判定条件；但此时第二关节（中间那个关节）到手肘的距离反而比指尖更远——这说明指尖实际上并没有伸到最前面。

**解决办法：** 增加一条额外的判断规则：**指尖到手肘的距离必须同时大于第二关节到手肘的距离**。这样就能把"手指弯了一半"的情况过滤掉，只有真正伸直的手指才会被认可。

### （六）"确认动作"换了三次才定下来

为了让用户选完楼层后能"确认"，我们试了三套完全不同的方案，前两个都被否决了：

| 第几代 | 方案 | 测试结果 | 为什么被否 |
|--------|------|---------|-----------|
| 第一代 | 食指往前戳一下 | 深度值（Z轴）忽大忽小，抖动太厉害 | 根本检测不出稳定的"戳"动作 |
| 第二代 | 用食指和拇指捏一下（比"OK"） | 捏的时候拇指靠近食指，但数字 5 的拇指判定逻辑也是"远离/靠近"，两者会打架 | 和数字 5 的判定条件冲突 |
| 第三代 | **五个指尖并拢收拢** | 自然、直观，看起来和任何数字手势都不一样 | ✅ **最终采用** |

### （七）"五指并拢"和数字 4 长得太像了

**遇到的问题：** 确认动作用"五指并拢"定下来之后，我们又发现新问题：数字 4 是四指伸直、大拇指弯曲，从正面看，五个指尖的展开范围和"五指并拢"非常接近，用一个简单的距离阈值根本分不开。

**排查过程：** 一开始我们想得很简单：只要五个指尖离得足够近，就认为是并拢。但看了数据才发现：
- 数字 4 的五指尖最大跨度是手掌宽度的 **1.00 ~ 1.08 倍**；
- 真正并拢时的五指尖最大跨度是手掌宽度的 **0.40 ~ 1.05 倍**。

这两个范围高度重叠，就像两个班的成绩单混在一起，怎么划线都会错判一部分。

**深入观察：** 我们不死心，把数据拆开看，终于发现了关键区别：
- 数字 4 的**大拇指是远离其他四指的**（因为大拇指弯曲了）；
- 并拢时**大拇指是紧贴其他四指的**（五个手指都收拢了）。
- 另外，数字 3 的四指本身就比较分散，而并拢时四指是紧紧聚在一起的。

**解决办法：** 我们不再只靠"一个指标"来判断，而是同时看**两个特征**：
1. **大拇指靠得近吗？** 拇指尖到其他任意一个指尖的距离要小于手掌宽度的 0.65 倍；
2. **四指聚得拢吗？** 食指到小拇指这四个指尖的最大跨度要小于手掌宽度的 0.30 倍。

只有两个条件**同时满足**，才认为是确认动作。这样数字 4（大拇指远）和数字 3（四指散）都被排除了。在 401 条真实数据中，数字 1~5 一次都没有被误触发，96 次真正的并拢动作也全部被正确识别，**准确率达到了 100%**。

### （八）并拢手指时，系统说"手心没朝我"

**遇到的问题：** 用户明明手心对着摄像头做了并拢动作，但系统偶尔不理睬，状态栏显示"手背朝向，已忽略"。

**排查过程：** 我们在代码里加了一个"调试打印"，把系统判断"手心朝向"时用的那个数值（cross_z）实时显示出来。结果发现：当五指并拢时，手指都聚在一起了，算出来的数值比手指张开时要小。原来设定的"及格线"（阈值 0.02）太严格了，导致一些合格的并拢动作被当成了"手背"。

**解决办法：** 我们把及格线从 **0.02** 逐步放宽到 **0.014**。这样既不会让真正的手背误通过，又能让并拢手势顺利通过检测。

### （九）选好楼层后，确认时楼层突然变了

**遇到的问题：** 用户先比了一个"3"，再并拢手指确认，结果电梯去的不是 3 楼，而是变成了 5 楼。

**排查过程：** 我们打开系统的"内部记忆"（8 帧环形缓冲区）观察，发现两个原因：
1. 并拢的瞬间，系统暂时识别不出数字，会标记为"-1（无效）"。这些"-1"进入缓冲区后，把之前好不容易确认下来的"3"给冲淡了；
2. 从"3"过渡到"并拢"的过程中，中间帧可能被系统看成"5"，导致缓冲区内混进了不一致的数字。

**解决办法：** 给防抖机制打了两个"补丁"：
1. **无效帧不写入记忆**："-1"这种无效结果不再进入缓冲区，之前确认好的数字就不会被冲掉；
2. **记忆一旦混乱就清空**：如果缓冲区内出现了不一致的数字（比如前面是 3，后面突然变成 5），立即把整个缓冲区清空，不让过渡帧捣乱。

### （十）电梯动画看不清

**遇到的问题：** 右侧的电梯井道动画刚做出来的时候，楼层数字特别小，线条也很细，在旁边观看演示的人根本看不清电梯到了哪一层。

**解决办法：** 我们对动画做了三项"视觉升级"：
1. 把楼层数字的字号**放大**，并且**加粗**；
2. 把井道的外框线和楼层分隔线都**加粗**，让对比度更明显；
3. 当电梯到达目标楼层时，该楼层的数字变成**青色高亮**，旁边还多了一个**闪烁的小圆点**作为提示。

改完之后，即使站得比较远，也能一眼看出电梯当前停在哪一层。

## 六、结论与展望

### （一）研究结论

本项目成功设计并实现了一套基于 MediaPipe 手势识别的无接触智慧电梯控制器。通过引入双特征联合判定算法、手势防抖机制和平移动画，系统在保证识别准确率的同时，提供了直观、流畅的用户体验。项目的主要创新点包括：

1. **双特征联合判定确认算法**：结合"拇指靠近度"与"四指离散度"，解决了传统单特征方法无法区分"数字 4"与"并拢确认"的难点；
2. **基于真实数据的阈值校准**：自主开发数据采集工具，采集 401 条样本进行统计分析，将阈值从经验猜测提升到数据驱动；
3. **电梯移动动画与中文 HUD**：在纯软件层面实现了媲美商用电梯的交互反馈，降低了部署成本；
4. **全模块化架构**：手势识别、状态控制、动画渲染、界面显示各模块解耦，便于后续扩展。
5. **开源托管**：项目完整代码与文档已上传至 GitHub（https://github.com/magiclab2233/gesture4lift），便于复现与二次开发。

### （二）不足与展望

尽管本项目在实验室环境下表现良好，但仍存在以下不足与改进空间：

1. **光照鲁棒性**：当前系统在强光或逆光环境下的识别稳定性有待提升，未来可引入图像预处理（如直方图均衡化）改善；
2. **多手支持**：目前仅支持单手识别，未来可扩展为双手同时识别，实现更复杂的交互逻辑（如双手"关门"手势）；
3. **硬件部署**：当前运行于 PC 端，未来可移植至树莓派或 Jetson Nano 等嵌入式平台，并接入真实电梯的串口或 MQTT 协议；
4. **语音反馈**：可加入 TTS（文本转语音）模块，为视障人士提供语音提示，提升系统的无障碍性。
